## 🏛️ UI ARCHITECTURE: THE FRONT GALLERY

The Assets Repository (Frontier Protocol) uses a high-density grid layout combined with a reactive **Details Panel** and a **Category Sidebar**.

### 1. Structural Layers
- **Sidebar (`sidebar`)**: Handles top-level navigation via `category-list`. It supports a "Floating HUD" aesthetic with Lucide icons.
- **Header (`repo-header`)**: Contains the Search Suite and Sort Suite. It dynamically switches to `compact-mode` when the Details Panel is active.
- **Grid (`assets-grid`)**: Fluid container for `asset-card` elements, rendered via `assets_gallery.js`.

### 2. The Reactive Details Panel
The `details-panel` implements an **Accordion UX** via `assets_details.js` ensuring high-density info display.
- **States**: `.collapsed` (applied to `.details-section`) vs `active`.
- **Memory**: `lastActiveSectionId` (Default: `section-preview`) persists the user's preferred detail section as they switch between assets.
- **Orchestration**: `resetDetailSections()` (in `assets_core.js`) resets the view to the primary preview for every fresh asset load to prevent visual stutter.

## ⚙️ LOGIC ENGINE: FILTER & SORT

The repository logic is managed by two primary engines:

### 1. The Gallery Engine (`assets_gallery.js`)
- **Fuzzy Filtering**: Combines `currentSearchTerm` and `currentCategory` to toggle `display: flex/none` on cards.
- **Multi-Sort Suite**: A real-time sorting engine using `getSortValue()` to handle 30+ criteria (Risk, Size, Compliance, Performance, Version).
- **DOM Re-insertion**: Uses `appendChild` to re-order existing DOM nodes based on the sort results, minimizing reflow costs.

### 2. The Data Manifest (`asset_manifest.js`)
- **Registry**: Large-scale JS array (`PROJECT_ASSETS`) providing the ground truth for names, types, sizes, and compliance status.
- **Integration**: `assets_core.js` initializes the gallery from this manifest on `DOMContentLoaded`.

### 3. Performance Constants (`assets_state.js`)
- **`MAX_PREVIEW_CHARS`**: 100,000 (~100KB) limit for text extraction to ensure DOM stability during decryption.
- **`HEAVY_FILE_THRESHOLD`**: **1,500,000 bytes** (1.5MB) for iframe/warning safety.

---

## 🚀 ASSET PREVIEW & HEAVY SIMULATION

The `assets_preview.js` engine handles the high-fidelity visualization of files.

### 1. Preview Logic
- **HTML/CSS**: IFRAME-based sandboxing for local code previews.
- **Images/Videos**: Native `<img>` and `<video>` tags with Frontier-themed overlays.
- **Logic/JSON**: Syntax-highlighted text blocks (Fetch-based).

### 2. Heavy Asset Protocol
Any file exceeding the threshold (1,500,000 bytes) triggers the **Heavy Loading State**:
- **Mechanism**: `createAutomatedLoadingState()` generates a simulated decryption sequence.
- **Simulation**: A complex CSS-animated progress bar with "Neural Data Initialization" labels to maintain the Lead Auditor aesthetic.

The Asset Repository is **COMPLIANT** with v2.9 Frontier standards. Its separation of concerns (Manifest vs Gallery vs Preview) allows for extreme scaling.
