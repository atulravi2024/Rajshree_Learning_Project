## 🔘 SEARCH MODE STATE MACHINE

The Navigation Hub operates in 3 distinct modes, each with its own execution and UI logic.

### 1. Mode Definitions (`window.SEARCH_MODE`)
- **`poi` (Point Interest)**: Focused on a single location.
    - **Trigger**: Clears existing paths and zooms to `150` units.
    - **Visual**: Only the destination coordinate is locked and labeled.
- **`route` (Direct Route)**: Start-to-Destination transition.
    - **Trigger**: Draws a 2-coordinate path.
    - **Visual**: Calculates the midpoint between points for camera focus.
- **`via` (Navigation Checkpoint)**: Multi-point routing.
    - **Trigger**: Draws a 3-coordinate path (`[fromCoords, viaCoords, toCoords]`).
    - **Visual**: Injects `viaInput` into the search logic.

### 2. UI Sync Logic (`updateSearchModeUI`)
- **Toggles**: Synchronizes `.mode-poi`, `.mode-route`, and `.mode-via` body classes for CSS layout shifts.
- **Dynamic Placeholders**: Updates input placeholders based on the active mode (e.g., "SEARCH CITY/NODE..." vs "START...").

## 📷 MIDPOINT CALCULATION ENGINE

A geometric helper ensuring the camera always focuses on the center of a route.

### Math Strategy
- **Base Midpoint**: `(lat1 + lat2) / 2`.
- **Longitude Wrap-Around**: 
    - Handles the **180-degree meridian wrap**.
    - If `Math.abs(lon2 - lon1) > 180`, the midpoint calculation adds `360` and then normalizes back into the `-180 to 180` range.
    - `midLon = (lon1 + lon2 + 360) / 2; if (midLon > 180) midLon -= 360;`

### Execution
1. Calculate `midLat`, `midLon`.
2. Invoke `window.rotateGlobeToCoords(midLat, midLon)`.
3. Force `window._mapTargetCameraZ = 150` for standard detail resolution.
