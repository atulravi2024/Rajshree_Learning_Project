## Domain-Driven Naming Convention Architecture

This markup represents an architecture that utilizes the structural benefits of the Monorepo model but replaces generic folder names (like `src/` and `tools/`) with explicit, domain-driven terminology.

This is the **ideal structure** if the primary goal is to make the folder layout instantly understandable to any beginner or AI agent.

### Why is this the Best Approach for Readability?
1. **Pristine Root Directory:** Your root folder is strictly reserved for global project configuration and GitHub metadata, exactly as it should be.
2. **Explicit Domain Separation:** Instead of wondering what belongs in `src`, developers immediately know that `frontend` contains the live Web App, and `backend` contains the server/infrastructure code.
3. **Dedicated Action Folders:** Isolating `database`, `scripts` and `tools` means the project scales cleanly—everyone knows exactly where data models, python automations, and utility libraries belong.
4. **Restored GitHub Native Rendering:** `README.md` and `CONTRIBUTING.md` are restored to their rightful place in the root directory, fixing the GitHub native rendering issues.
