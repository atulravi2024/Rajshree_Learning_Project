# Memory Folder Architecture

The `memory/` folder acts as your **AI Development Context Hub**. It is explicitly designed to serve as the "brain text" for the LLM agents and human developers maintaining the project. 

It never contains live production code. Instead, it holds the knowledge, rules, boundaries, and structural blueprints of the application so context is never lost across sessions.

---

## 📂 The Ultimate Structure

```text
memory/
├── STRICT_BOUNDARIES.md         <-- Global Rules: Absolute constraints the AI/Devs must never break.
├── MEMORY_ARCHITECTURE.md       <-- This file: Explains how this entire folder is organized.
│
├── current/                     <-- Live State Tracking (The "Now")
│   ├── PROJECT_HTML.md          <-- Live map of the DOM structure, IDs, and templates.
│   ├── PROJECT_CSS.md           <-- Live map of the styling, themes, and global variables.
│   ├── PROJECT_JS.md            <-- Live map of scripts, logic, routing, and functions.
│   ├── PROJECT_ASSETS.md        <-- Live map of audio limits, naming conventions, and images.
│   └── PROJECT_MEMORY.md        <-- Central hub linking all the above and immediate goals.
│
├── research/                    <-- Deep Dives & Data Schemas
│   ├── Database/                <-- Documentation on original database structures.
│   ├── Expansion/               
│   │   └── DATABASE_STRUCTURE_250_COLS.md  <-- Massive database schemas and feature planning.
│   ├── Roadmap/                 <-- Long-term planning, phases, and future feature targets.
│   └── TTS_Engines/             <-- Research and configuration notes for Text-to-Speech models.
│
├── 03 Architecture & Planning/      <-- Blueprints & Overhauls
│   ├── Current_Architecture.md             <-- Where the project is today.
│   ├── Proposed_Monorepo_Architecture.md   <-- Where it could go tomorrow.
│   └── Ultra_Refined_Modular_Architecture.md
│
├── demo/                        <-- Temporary scratchpad for generating quick proofs-of-concept.
│
└── github/                      <-- Drafts specific to CI/CD, Actions, or Repository logs.
```

---

## 🧠 Core Philosophy
1. **The Source of Truth:** If an agent forgets how the CSS works, it is instructed to read `01 Current State/01.02_PROJECT_CSS.md`. These files must be updated whenever the code changes.
2. **Immutable Rules:** `STRICT_BOUNDARIES.md` sits at the very root of `/memory` so it is always read first before doing any work.
3. **Drafting:** Experimental plans and massive schemas (like your 250 columns) go into `research/` or `03 Architecture & Planning/`. This ensures they never pollute the `current/` state files until the features are actually built.

This structure guarantees that no matter how massive the **Rajshree Learning Project** becomes, the AI assisting you will instantly understand the entire architecture on day 1 or day 1000.
