# Hybrid: Johnny Decimal + Domain-Driven Memory Architecture

This document defines a hybrid numbering system for organizing the `memory/` folder. It combines the structured rigidity of the **Johnny Decimal System** with the logical boundaries of **Domain-Driven Design (DDD)**. 

This approach uses a two-level decimal system (`[Category].[ID]`) where Categories map to distinct Domains of the project.

## Domain Breakdown (Johnny Decimal Categories)

### 00: System & Guardrails Domain
This domain holds the immutable meta-rules and index files that govern the entire repository and the mind of the AI.
*   **`00.01_STRICT_BOUNDARIES.md`**: Absolute guardrails and constraints. Must be read first.
*   **`00.02_MEMORY_ARCHITECTURE.md`**: (This file) The blueprint explaining how the memory folder is structured.

### 01: Current State Domain
The source of truth for the living application. This domain maps the active, production-ready project files.
*   **`01.01_PROJECT_HTML.md`**: Live map of DOM structure, IDs, and templates.
*   **`01.02_PROJECT_CSS.md`**: Live map of styling, themes, animations, and external libraries.
*   **`01.03_PROJECT_JS.md`**: Live map of scripts, routing, state management, and functions.
*   **`01.04_PROJECT_ASSETS.md`**: Live map of fonts, audio parameters, edge-tts files, and images.
*   **`01.05_PROJECT_MEMORY.md`**: Central hub connecting the current state to immediate goals.

### 02: Research & Knowledge Domain
This domain holds deep dives, external model research, and data structures. It never pollutes the Current State.
*   **`02.01_Database_Schemas/`**: Documentation on database structures (e.g., 250-column layouts).
*   **`02.02_TTS_Engines/`**: Research, configuration notes, and bounds for Edge-TTS and other models.
*   **`02.03_Roadmap/`**: Long-term planning, phases, and feature targets.

### 03: Architecture & Planning Domain
Blueprints for project restructuring, overhauls, and directory reorganizations.
*   **`03.01_Current_Architecture_Map.md`**: How the project outside the memory folder currently looks.
*   **`03.02_Proposed_Architecture/`**: Future plans (e.g., Monorepo transitions, Domain-Driven project structures).

### 04: Sandbox & Demos Domain
Temporary isolation zones for experimental proofs-of-concept.
*   **`04.01_DEMO_ASSETS.md`**: Tracks isolated assets for testing.
*   **`04.02_README.md`**: Explains the rationale for the isolation zone.

### 05: Infrastructure & Workflows Domain
Documentation related to CI/CD, deployments, and repo management.
*   **`05.01_GITHUB_CHANGELOG.md`**: Tracks changes across the GitHub repo setup.
*   **`05.02_GITHUB_MEMORY.md`**: Explains GitHub workflows, deployment rules, and the `frontend/` project file structure.

---

### Core Philosophy of the Hybrid Method
1.  **Strict Ordering:** The `00` domain is always processed first ensuring boundaries are respected before any development begins.
2.  **Domain Segregation (DDD):** A change in the CSS (`01`) does not require reading the Roadmap (`02`), significantly reducing context pollution.
3.  **Future Expansion (Johnny Decimal):** To add a new domain (e.g., Unit Testing context), you simply allocate the next available category prefix (e.g., `06 Testing`).
4.  **Item Identification:** Every single file and folder has a unique, absolute identifier (e.g., `01.02`) making it instantly referencable and unmistakable.
